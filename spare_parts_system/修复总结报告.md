# 物料号匹配逻辑修复总结报告

## 📋 问题诊断

### 根本原因
1. **采购表"原物料号"列全是NaN**：原代码在装箱关联时错误地使用了"原物料号"列（全是nan），导致无法匹配
2. **箱号表加载失败**：DataEngine扫描文件时未过滤Excel临时文件（~$开头），导致读取失败
3. **物料号列引用丢失**：主表在清理临时列时删除了物料号引用，导致后续装箱关联无法获取正确的物料号

## 🔧 修复方案

### 1. 修复 `build_chain_master_v2` 函数（sales.py）

#### 修复点 1：保存物料号源列引用
```python
# 【新增】保存原始物料号列的引用，用于后续装箱关联
master_df["_物料号源列"] = part_col
```

**作用**：保留缺货表原始物料号列的列名，供后续装箱关联使用

#### 修复点 2：修改装箱关联逻辑使用正确的物料号列
```python
# 【修复】使用保存的物料号源列来获取物料号
part_col_name = row.get("_物料号源列", None)
if part_col_name and part_col_name in master_df.columns:
    part_no = strict_clean(row.get(part_col_name, ""))
else:
    # 降级方案：尝试使用match_key
    if "match_key" in master_df.columns:
        part_no = row.get("match_key", "")
    else:
        part_no = ""
```

**作用**：从主表的原始物料号列获取物料号，而不是使用已删除的临时列

#### 修复点 3：统一使用 strict_clean 清洗关联键

**采购表关联键清洗：**
```python
# 【修复】使用strict_clean清洗SAP订单号和物料号作为关联键
proc_df["_SAP订单号"] = proc_df[sap_order_col].apply(strict_clean)
proc_df["_match_key"] = proc_df["物料号"].apply(strict_clean)
```

**缺货表关联键清洗：**
```python
# 统一使用 strict_clean 清洗物料号
master_df["match_key"] = master_df[part_col].apply(strict_clean)
```

**作用**：确保采购表、箱号表、缺货表使用相同的清洗规则（去空格、去前导零、去.0），提高匹配成功率

### 2. 修复 `DataEngine.get_all_data_files` 方法（data_engine.py）

#### 修复点 4：过滤Excel临时文件
```python
# 过滤Excel临时文件（以~$开头）
if file_path.name.startswith('~$'):
    continue
```

**作用**：防止DataEngine扫描到Excel打开时产生的临时文件，避免读取失败

## 📝 关键修复点总结

### 1. 确保 strict_clean 应用于所有关联键
- ✅ 采购表的 SAP订单号
- ✅ 采购表的 物料号
- ✅ 箱号表的 SAP需求单号

- ✅ 缺货表的 物料号

### 2. 双因子匹配逻辑（SAP订单号 + 物料号）
采购表 ➡️ 装箱表的关联现在正确使用：
- **因子1**：SAP订单号（采购表） = SAP需求单号（箱号表）
- **因子2**：物料号（采购表） = 物料号（箱号表）

两个因子都必须匹配才能成功关联。

### 3. 预处理采购表逻辑保持不变
`preprocess_procurement_table` 函数仍然会优先使用"原物料号"（如果有值），但在本数据集中，"原物料号"大部分为空，所以会降级使用"物料号"列。
